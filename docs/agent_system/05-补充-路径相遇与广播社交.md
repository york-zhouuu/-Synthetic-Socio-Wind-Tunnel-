**本文档补充 01-04 的设计，解决两个关键问题。**

## 问题 1: 路上相遇

### 现状的问题

当前系统中 agent 在 location 之间是瞬间跳跃的：

```
tick 10: Emma at home
tick 11: Emma at sunrise_cafe     ← 瞬移！中间经过的 main_street 不存在
```

这意味着两个人走同一条路永远不会碰面，不自然。

### 解决方案：逐步移动 (Step-by-Step Movement)

**Agent 按照导航路径逐步移动，每个 tick 移动一步。**

NavigationService 已经返回完整路径：

```python
route = nav.find_path("home", "sunrise_cafe")
# route.steps = [
#     Step(location="home", action="exit_building"),
#     Step(location="residential_street", action="walk"),     ← 在路上
#     Step(location="main_street", action="walk"),            ← 在路上
#     Step(location="sunrise_cafe", action="enter_building"),
# ]
```

Orchestrator 每个 tick 只推进一步：

```python
# 不是这样（瞬移）:
simulation.move_entity("emma", "sunrise_cafe")

# 而是这样（逐步）:
class AgentRuntime:
    def __init__(self):
        self.movement_queue: list[str] = []  # 待经过的 location 序列

    def start_moving(self, route: NavigationResult):
        """开始沿路线移动。"""
        self.movement_queue = [step.location_id for step in route.steps]

    def tick_move(self, simulation) -> str | None:
        """每个 tick 移动一步。"""
        if not self.movement_queue:
            return None
        next_location = self.movement_queue.pop(0)
        simulation.move_entity(self.agent_id, next_location)
        return next_location
```

**效果：**
```
tick 10: Emma at home                    Bob at office
tick 11: Emma at residential_street      Bob at office_street
tick 12: Emma at main_street             Bob at main_street     ← 在路上碰到了！
tick 13: Emma at sunrise_cafe            Bob at central_park
```

### 对地图的影响

**道路必须作为 OutdoorArea 存在于地图中**，不能只是 Connection。

这和地图构建方案中的 Connection 推断需要调整：

```
之前的思路:
  Building A ──Connection──> Building B
  (道路只是两个建筑之间的连线)

现在的思路:
  Building A ──Connection──> Street Segment ──Connection──> Building B
  (道路本身是一个 OutdoorArea，agent 会经过它)
```

在 GeoJSON Importer 中：
- OSM 的道路 (highway) 不仅用于推断 Connection
- **道路本身也要作为 OutdoorArea 导入**（surface="asphalt", area_type="street"）
- 长道路可以分段（每 50-100m 一段），这样不会所有人都在同一个巨大的"道路 location"里

```python
# 道路分段策略
def _segment_road(self, road_points: list[Coord], segment_length: float = 80.0) -> list[OutdoorArea]:
    """将一条长道路分为多个 OutdoorArea 段。"""
    segments = []
    current_segment = [road_points[0]]

    for i in range(1, len(road_points)):
        current_segment.append(road_points[i])
        segment_dist = sum(
            current_segment[j].distance_to(current_segment[j+1])
            for j in range(len(current_segment)-1)
        )
        if segment_dist >= segment_length:
            # 创建一个街道段
            segments.append(self._create_street_segment(current_segment, road_name))
            current_segment = [road_points[i]]  # 新段从当前点开始

    return segments
```

### 移动速度

不同道路类型的移动速度不同（每 tick 经过的 location 数）：

| 道路类型 | 速度 | 说明 |
|---------|------|------|
| footway | 1 segment/tick | 步行，最慢 |
| residential | 1 segment/tick | 住宅区道路 |
| primary | 2 segments/tick | 主干道，走得快或有公交 |
| path (公园内) | 1 segment/tick | 散步速度 |

## 问题 2: 广播式社交

### 设计思路

**社交不是 1:1 配对，而是基于"位置广播"。**

每个 tick，每个 location 就像一个"房间"——里面所有人都能感知到彼此的存在和行为。

### 社交事件模型

```python
class SocialSignal(BaseModel):
    """一个社交信号（广播）。"""
    signal_id: str
    emitter_id: str                  # 发出者
    location_id: str                 # 发生地点
    signal_type: str                 # "greeting" / "conversation" / "activity" / "broadcast"
    content: str                     # 信号内容
    volume: float                    # 0-1, 影响传播范围
    target_id: str | None = None     # 定向信号 (私聊) or None (广播)

class SocialContext(BaseModel):
    """某个 location 在某个 tick 的社交环境。"""
    location_id: str
    tick: int
    agents_present: list[str]        # 在场的所有 agent
    active_signals: list[SocialSignal]  # 当前活跃的社交信号
    ongoing_conversations: list[str]    # 正在进行的对话 ID
```

### 广播社交的执行流程

```
每个 tick 的社交处理:

1. 构建 SocialContext
   ├── 按 location 分组所有 agent
   └── 每个 location 生成一个 SocialContext

2. 生成社交信号
   ├── 对每个 location 中的每个 agent:
   │   └── 根据性格和当前状态，决定是否发出信号
   │       (外向的人更容易主动说话)
   └── 信号添加到 SocialContext.active_signals

3. 广播信号接收
   ├── 同一 location 的所有 agent 都收到所有信号
   ├── 每个 agent 根据自己的性格/兴趣决定是否回应
   └── 信号也可以传播到相邻 location（通过 audible_range）

4. 对话形成
   ├── 如果 agent A 发出信号，agent B 回应了 → 对话开始
   ├── agent C 听到了对话，也加入 → 群体对话
   └── 对话是一个"持续实体"，可以跨多个 tick
```

### 对话机制（LLM 间真正的交流）

```python
class Conversation:
    """一个正在进行的对话。"""

    def __init__(self, location_id: str, participants: list[str]):
        self.conversation_id: str = uuid()
        self.location_id: str = location_id
        self.participants: list[str] = participants
        self.messages: list[ConversationMessage] = []
        self.is_open: bool = True     # 其他人可以加入

    def add_message(self, speaker_id: str, content: str):
        self.messages.append(ConversationMessage(
            speaker=speaker_id,
            content=content,
            tick=current_tick,
        ))

    def is_joinable(self) -> bool:
        """其他人能否加入这个对话。"""
        return self.is_open and len(self.participants) < 6  # 群聊上限
```

### LLM 对话生成

**关键：对话是多方的，不是 1:1。**

```python
async def run_conversation_round(
    conversation: Conversation,
    agents: dict[str, AgentRuntime],
    llm_client,
) -> list[ConversationMessage]:
    """
    运行一轮对话。

    每个参与者根据自己的视角和记忆，决定说什么（或沉默）。
    """
    new_messages = []

    for participant_id in conversation.participants:
        agent = agents[participant_id]

        # 只有 LLM 驱动的 agent 才生成对话
        if agent.profile.tier == "background":
            continue

        # 构建这个 agent 视角的对话上下文
        prompt = f"""
你是 {agent.profile.name}。
你的性格: 外向性={agent.profile.extroversion}, 兴趣={agent.profile.interests}
你的相关记忆: {agent.memory.retrieve(conversation.topic, k=3)}

你正在 {conversation.location_id}，以下是当前对话:
{format_messages(conversation.messages[-10:])}  # 最近 10 条

在场的其他人: {conversation.participants}

请决定:
1. 你要说什么？(或者保持沉默，输出 [SILENT])
2. 你从这段对话中了解到了什么新信息？

输出 JSON:
{{"response": "你要说的话 或 [SILENT]", "learned_info": ["信息1", "信息2"]}}
"""

        result = await llm_client.generate(prompt, model=agent.profile.model)

        if result.response != "[SILENT]":
            msg = ConversationMessage(speaker=participant_id, content=result.response)
            new_messages.append(msg)
            conversation.add_message(participant_id, result.response)

        # 记录学到的新信息到记忆
        for info in result.learned_info:
            agent.memory.record_event(MemoryEntry(
                memory_type="event",
                content=f"从 {conversation.get_speaker_name()} 处听说: {info}",
                importance=0.6,  # 社交获得的信息重要性中等
                related_agents=[m.speaker for m in conversation.messages],
            ))

    return new_messages
```

### 信息传播链

这是广播式社交最重要的效果——**信息可以在 agent 网络中自然传播**：

```
Tick 100: 系统推送信息给 Emma → "公园今晚有活动"
          Emma 记忆: {content: "公园今晚有活动", source: "phone"}

Tick 110: Emma 在咖啡馆遇到 Bob，对话中提到了这件事
          Bob 记忆: {content: "从 Emma 处听说公园今晚有活动", source: "Emma"}

Tick 130: Bob 路上遇到 Chen，提到了公园的事
          Chen 记忆: {content: "从 Bob 处听说公园今晚有活动", source: "Bob"}
          → 信息传播了两跳！

Tick 150: Emma, Bob, Chen 都去了公园 → 社交碰撞概率大增
          → 这就是 Policy Hack "信息注入" 的涌现效果
```

**这条链可以被追踪和可视化：**

```json
{
  "info_chain": {
    "origin": "system_injection",
    "content": "公园今晚有活动",
    "hops": [
      {"from": "system", "to": "emma", "tick": 100, "channel": "phone"},
      {"from": "emma", "to": "bob", "tick": 110, "channel": "conversation", "location": "sunrise_cafe"},
      {"from": "bob", "to": "chen", "tick": 130, "channel": "conversation", "location": "main_street"}
    ]
  }
}
```

### 社交互动的分层处理（成本控制）

| 参与者组合 | 处理方式 | LLM 调用/轮 |
|-----------|---------|-------------|
| Protagonist × Protagonist | 完整多轮对话 (3-5 轮) | 2-5 次/轮 × 3-5 轮 |
| Protagonist × 任何 agent | 对方临时升级，对话 (1-3 轮) | 2 次/轮 × 1-3 轮 |
| 非 Protagonist × 非 Protagonist | 1 次 LLM 生成对话摘要 | 1 次 |
| 3+ agent 群聊 (含 Protagonist) | 多轮广播式 (每轮每人一次) | N×轮数 次 |

**预估成本调整（加入社交对话后）：**

| 场景 | 原估算 (无对话) | 新估算 (有对话) |
|------|----------------|----------------|
| 每模拟日 LLM 调用 | ~360 次 | ~500-800 次 |
| 用 Haiku 的成本 | $0.5-2.0 | $1.0-4.0 |

增加不多，因为大部分社交发生在 Active-Active（1 次调用）或涉及 Background（0 次调用）。
