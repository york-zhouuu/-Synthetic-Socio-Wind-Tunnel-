# Fitness Audit 报告解读

`data/fitness-report.json` 是 `make fitness-audit` 产出的结构化报告。本文
解释每条 AuditResult 怎么读，以及如何把它作为 Phase 2 change 的动机锚点。

## 报告结构

```
FitnessReport
├── schema_version        "1.0"
├── generated_at          ISO datetime
├── atlas_source          "data/lanecove_atlas.json"
├── atlas_signature       SHA-256 of the atlas JSON
├── categories            8 类审计结果
├── scale_baseline        wall-time 快照
├── cost_baseline         LLM 成本估算范围
├── site_fitness          场地-thesis 适配度诊断
└── seeds                 复现种子
```

每条 `AuditResult`：

```
{
  "id": "e1.push-reaches-target",
  "status": "pass" | "fail" | "skip",
  "detail": "...",
  "mitigation_change": "attention-channel" | "memory" | ...,  # for fail/skip
  "extras": {...}  # 可选的数值元数据
}
```

## 十类审计（分三组）

### 基线存在性（核心锚点来源）

| category | 目的 | 典型 fail 条目 → mitigation |
|---|---|---|
| `phase1-baseline` | 用 import / 字段集 探针检查"本 change 及之前的能力表面是否存在" | preset-ground-truthed 永远 fail → `agent` |
| `phase2-gaps` | 对每块 Phase 2 capability 探测模块路径；未建成 → fail | 7 条 fail 对应 7 个 Phase 2 cap |

基线探针不调用新代码的任何功能——只查"这个模块 / 这个字段 / 这个枚举值
存不存在"，避免"自己写代码 → 自己测试代码 → 自己合理化"的套娃。

### 集成（假设能力已存在，测端到端）

| category | 覆盖 | 失败 → mitigation |
|---|---|---|
| `e1-digital-lure` | 推送可达性、注意力抑制、日志可导 | `attention-channel` |
| `e2-spatial-unlock` | 运行期 unlock_door、路径差异、事件日志 | `cartography` / `navigation` |
| `e3-shared-perception` | 共享 looking_for、共享任务持久化 | `memory` / `perception` |
| `profile-distribution` | 1000 样本覆盖结构维度 | `agent` / `attention-channel` |
| `ledger-observability` | 轨迹、相遇、决定性 | `ledger` / `metrics` |

### 诊断（纯数据，不 gate）

| category | 指标 | 说明 |
|---|---|---|
| `site-fitness` | 建筑名/密度/住宅比 | Lane Cove 是确定场地，不跨场地对比 |
| `scale-baseline` | quick 100×72 / full 1000×288 wall time | 每 tick 真正做 move + render（不含 LLM） |
| `cost-baseline` | 每日 USD 估算 | **硬编码参数，未经测量**——见 `cost.py` 顶部注释 |

## 用法：Phase 2 change 的前置证据

每块 Phase 2 change 在开工前：

1. 跑 `make fitness-audit`，刷新 `data/fitness-report.json`
2. 在 change proposal 的 `## Why` 中，**至少引用一条**该能力对应的
   `fail` 或 `skip` AuditResult，格式：

   > 来自 `data/fitness-report.json` 中的 `e3.shared-task-memory-seam`（skip，
   > `mitigation_change="memory"`）：Phase 1 无持久化 per-agent task 存储，
   > 本 change 通过三层 memory 结构填补此缺口。

3. 如果无对应条目（例：`social-graph` 目前没有 skip/fail 指向它），
   proposal 应自己论证 thesis 层面的动机，同时建议为后续审计添加该条目
   （例：扩展 `e3-shared-perception` 检查"关系边在 ledger 中是否可查询"）。

## 刷新与提交策略

- 审计报告 `data/fitness-report.json` 不进 git（已在 `.gitignore`），
  是"某一时间点的体检"，版本化会产生无意义的 diff；
- 每次 Phase 2 change 开工前**重新跑一次**，以当时的 atlas/代码为基准；
- 审计代码本身在 `synthetic_socio_wind_tunnel/fitness/`，改动会影响所有
  后续审计的语义——若需改审计逻辑，先开独立 change。

## 与 OpenSpec 的关系

- `fitness-audit` 作为一级 capability 定义在
  `openspec/specs/fitness-audit/spec.md`（由 `realign-to-social-thesis`
  change archive 后生成）
- 审计代码每条 SHALL 要求都可以追到 spec 中的一条 scenario
- 审计失败不阻塞开发；只阻塞"在未解释缺口的情况下做 Phase 2"
