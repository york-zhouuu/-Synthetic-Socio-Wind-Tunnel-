# Agent 系统：代码改动清单与执行计划

**本文档供下次 Claude Code 会话使用：打开此文件即可理解完整上下文并开始编码。**

---

## 背景摘要

synthetic_socio_wind_tunnel 是一个社会模拟引擎，已有完整的地图(Atlas)、状态(Ledger)、模拟(Simulation)、感知(Perception)基础设施。现在需要在此之上构建 Agent 系统——让 entity 具备自主决策、记忆、社交能力，并支持 Policy Hack 干预实验。

目标规模：1000 agent（分三层：Protagonist 用贵模型、Active 用便宜模型、Background 用规则引擎）。

---

## 新增文件清单

所有新文件在 `synthetic_socio_wind_tunnel/agent/` 目录下：

```
synthetic_socio_wind_tunnel/agent/
├── __init__.py              # 公共 API 导出
├── profile.py               # AgentProfile: 性格、身份、习惯
├── memory.py                # AgentMemory: 三层记忆系统
├── planner.py               # Planner: LLM 计划生成 + 重规划
├── social.py                # SocialGraph: 关系网络
├── orchestrator.py          # Orchestrator: 模拟主循环调度
├── model_budget.py          # ModelBudget: 动态模型等级分配
├── intervention.py          # Intervention: Policy Hack 干预机制
├── metrics.py               # MetricsCollector: 实验指标收集
└── runtime.py               # AgentRuntime: 单个 agent 的运行时封装
```

## 对现有代码的改动

### 改动极少，核心原则：不改现有模块的内部逻辑

| 文件 | 改动 | 原因 |
|------|------|------|
| `synthetic_socio_wind_tunnel/__init__.py` | 新增 agent 模块的导出 | 公共 API |
| `synthetic_socio_wind_tunnel/ledger/service.py` | 可能增加 `get_entities_at_location()` 方法 | 社交碰撞检测需要按 location 查询 entity |

**不改动的文件：**
- `atlas/` — 完全不动
- `engine/simulation.py` — 完全不动（agent 通过调用现有 API 执行动作）
- `engine/navigation.py` — 完全不动
- `perception/` — 完全不动（agent 通过调用现有 API 获取感知）
- `cartography/` — 完全不动

## 执行顺序（推荐分 4 个 Phase）

### Phase 1: 基础框架 (可独立运行的最小可行版本)

```
目标: 1 个 agent 能在地图上按 LLM 生成的计划移动

新建文件:
  ├── agent/__init__.py
  ├── agent/profile.py          # AgentProfile 数据模型
  ├── agent/planner.py          # LLM 日计划生成 (用 Haiku)
  └── agent/runtime.py          # AgentRuntime 基础封装

验证:
  创建 1 个 AgentProfile
  → LLM 生成日计划
  → 按计划调用 SimulationService.move_entity() 逐步移动
  → 输出轨迹 JSON
```

### Phase 2: 记忆与 LLM 决策

```
目标: Agent 有四层记忆，能用 LLM 生成基于 Life Pattern 的计划

新建文件:
  ├── agent/memory.py           # AgentMemory 四层记忆 (含 Life Pattern)
  └── (更新) agent/planner.py   # 增加 LLM 计划生成，以 Life Pattern 为基础输入

验证:
  创建 1 个 Protagonist agent
  → 从 Profile 初始化 Life Pattern (Layer 4)
  → LLM 基于 Life Pattern 生成日计划
  → 移动过程中记录 Layer 1 事件
  → 日终生成 Layer 2 摘要和 Layer 3 反思
  → 第二天的计划会参考昨天的记忆
  → 连续跑 3 天后检查 Life Pattern 是否被行为更新
```

### Phase 3: 社交与多 Agent

```
目标: 多个 agent 能相遇并互动，模型等级动态分配

新建文件:
  ├── agent/social.py           # SocialGraph 关系网络
  ├── agent/model_budget.py     # 动态模型等级分配
  └── agent/orchestrator.py     # 多 agent 调度循环

改动:
  └── ledger/service.py         # 增加 get_entities_at_location()

验证:
  创建 5 个 agent (混合层级)
  → 运行 1 模拟日
  → 至少发生 1 次社交碰撞
  → SocialGraph 中产生新关系
  → 输出社交事件日志
```

### Phase 4: 干预与实验

```
目标: 完整的实验闭环

新建文件:
  ├── agent/intervention.py     # Policy Hack 干预
  └── agent/metrics.py          # 指标收集与对比

验证:
  配置一个 "数字诱饵" 实验
  → 跑 3 天基线 + 1 天干预 + 3 天观察
  → 输出:
     - 轨迹 JSON (前端可视化用)
     - 社交事件日志
     - 指标对比 (control vs treatment)
     - Protagonist 的叙事文本
```

## 关键接口约定

Agent 模块调用现有 synthetic_socio_wind_tunnel API 的方式：

```python
# 移动 agent
result = simulation.move_entity(agent_id="emma", to_location="sunrise_cafe")

# 获取感知
view = perception.render(ObserverContext(
    entity_id="emma",
    position=agent.position,
    location_id=agent.current_location,
    skills={"investigation": 0.3, "perception": 0.5},
    knowledge=agent.memory.get_knowledge(),
    emotional_state={"curiosity": 0.8},
))

# 寻路
route = navigation.find_path("home", "sunrise_cafe")

# 获取同一位置的其他 entity
entities_here = ledger.get_entities_at_location("sunrise_cafe")

# 推进时间
simulation.advance_time(minutes=5)
```

## 与地图构建 Pipeline 的关系

Agent 系统依赖地图数据，但两者可以并行开发：

```
地图构建 Pipeline                    Agent 系统
    │                                    │
    ├── Step 1: models.py 加字段          │
    │                                    ├── Phase 1: profile + planner (规则版)
    ├── Step 2: importer 增强             │   (用测试地图即可)
    │                                    │
    ├── Step 3: enricher 新建             ├── Phase 2: memory + LLM
    │                                    │
    ├── Step 4: 端到端测试                ├── Phase 3: social + orchestrator
    │                                    │
    └── 产出: atlas.json ──────────────→ Phase 4: intervention + metrics
                                              (需要真实地图)
```

Phase 1-3 可以用测试地图（`tests/conftest.py` 中已有的 fixture），Phase 4 才需要真实地图数据。
